# סבב סודות (Secret Rotation) — נהלים ורשימת ביצוע

נכתב בעקבות דוח האבטחה (ממצא #2): קובץ `.env` עם סודות אמיתיים היה חשוף לסיכון דרך ZIP
שיוצא ידנית מתיקייה חיה (ולא דרך git — `.gitignore` תמיד החריג אותו כראוי).

**חשוב:** מסמך זה הוא תיעוד/הכנה בלבד. אף סוד לא הוחלף בפועל בסביבת הייצור כחלק מפעימה 1 —
כי חלק מהמפתחות (IVR_KEY, TECHNOLINE_API_KEY) הם צד-שני אצל טכנוליין, והחלפתם בלי תיאום
תשבור שיחות IVR חיות. בצע את הרוטציה בפועל רק כשיש חלון תחזוקה מתואם.

## אילו מפתחות ב-.env דורשים רוטציה, ולמה כל אחד שונה

| מפתח | תלות חיצונית? | מה קורה בהחלפה | סדר מומלץ |
|---|---|---|---|
| `JWT_SECRET` | **לא** — פנימי לגמרי | כל המשתמשים המחוברים מנותקים מיידית (הטוקנים הישנים נפסלים) | הכי בטוח להחליף לבד, בלי תיאום חיצוני |
| `IVR_KEY` | **כן** — טכנוליין שולח אותו כ-`ivrKey=` ב-query על כל בקשת IVR | אם משנים כאן בלי לעדכן אצל טכנוליין — **כל שיחה חיה תיכשל** (403) עד שהם יעדכנו | לתאם מול טכנוליין מראש, להחליף בו-זמנית בשני הצדדים |
| `TECHNOLINE_API_KEY` | **כן** — משמש לשיחות Click-to-Call/קמפיינים יוצאים דרך API של טכנוליין | קמפיינים/חיוג ידני ייכשלו עד עדכון | לתאם מול טכנוליין |
| `CREDIT_CARD_TERMINAL` | **כן** — מזהה טרמינל סליקה (Nedarim Plus / טכנוליין) | תשלומי כרטיס אשראי ב-IVR ייכשלו | לתאם מול ספק הסליקה |
| `SESSION_TIMEOUT_HOURS`, `DB_PATH`, `PORT`, `NODE_ENV` | לא רלוונטי | אלה לא סודות | אין צורך ברוטציה |

## שלב 1 — אפשר לבצע כבר עכשיו, בלי תיאום (סיכון נמוך)

יצירת JWT_SECRET חדש (64 בתים אקראיים, אותה שיטה שכבר מתועדת ב-README_PRODUCTION.md):

```bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
```

עדכן את הערך ב-`.env` בסביבת הייצור (לא בקוד, לא ב-git), ואז:

```bash
pm2 restart donors
```

**תופעת לוואי צפויה:** כל מי שמחובר כרגע ינותק ויצטרך להתחבר מחדש. מומלץ לבצע בשעה שקטה.

## שלב 2 — דורש תיאום עם טכנוליין לפני ביצוע

1. צור ערך חדש: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`.
2. פנה לאיש הקשר בטכנוליין ותאם חלון זמן קצר להחלפת ה-IVR Key/API Key בצד שלהם.
3. רק לאחר אישור מהם שהם עדכנו — עדכן את `IVR_KEY` (ו/או `TECHNOLINE_API_KEY`) ב-`.env` בייצור ובצע `pm2 restart donors`.
4. ודא מיד אחרי: `curl "http://localhost:3000/ivr?ivrKey=<המפתח_החדש>&PBXphone=0521234567&PBXcallId=rotation-test"` מחזיר תשובת IVR תקינה (לא 403).
5. אם טכנוליין מתעדכנים במועד מאוחר יותר מכם — **אל תחליפו את הצד שלכם קודם**, אחרת כל שיחה חיה תיכשל בינתיים.

## שלב 3 — טרמינל אשראי

תיאום דומה מול ספק הסליקה (Nedarim Plus / טכנוליין creditCard) לפני עדכון `CREDIT_CARD_TERMINAL`.

## אחרי כל רוטציה

- לוודא ש-`.env` עדיין לא נכנס בטעות ל-git: `git status` בסביבת הייצור אחרי העדכון.
- לתעד בלוג הביקורת הפנימי (לא ב-git) מתי בוצעה הרוטציה ועל ידי מי.
- לשקול תזמון רוטציה תקופתית (לדוגמה כל 6-12 חודשים) גם בלי אירוע חשיפה, במיוחד ל-IVR_KEY שרץ כרגע על HTTP (ראו `nginx/ivr-site.conf`).

## אימות מוכנות (פעימה 4B, 2026-07-09)

לפני שממליצים לבצע רוטציה בפועל, אומת שהקוד עצמו לא יעמוד בדרך:

- **כל סוד בקוד מגיע מ-`process.env`, בלי יוצא מן הכלל** — נבדק בחיפוש מקיף על `JWT_SECRET`, `IVR_KEY`, `TECHNOLINE_API_KEY`, `CREDIT_CARD_TERMINAL`, `ALFON_SYNC_KEY`, `ALFON_API_URL` בכל קובצי הבקאנד. כולם קוראים מ-`process.env.X || ""` — אין שום ערך סוד קשיח (hardcoded) במקום כלשהו.
- **`ALFON_API_URL`** כן כולל fallback קשיח בקוד — אבל זה URL ציבורי (לא סוד), החלטה מתועדת מפעימה 2, לא רלוונטי לרוטציה.
- **`.github/workflows/deploy.yml`** (קובץ פריסה אוטומטית שקיים ב-git אך מוגדר כ"נדחה" לפי הזיכרון הפנימי) נבדק גם הוא — משתמש אך ורק ב-GitHub Secrets (`secrets.SERVER_IP`/`SERVER_USER`/`SSH_PRIVATE_KEY`), אין שם שום ערך קשיח.
- **מסקנה: הקוד מוכן לרוטציה בכל רגע** — אין מקום בקוד שדורש שינוי לפני שמחליפים את הערכים ב-`.env` בפרודקשן. השלב היחיד שנשאר הוא תיאום אנושי מול טכנוליין/ספק הסליקה (שלבים 2-3 למעלה) ובחירת חלון תחזוקה.
