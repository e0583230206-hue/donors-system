# הקלטות systemMessages למודול הסליקה — S3000–S3023
## גרסה 1.0 — טיוטת הכנה (לא סופי, לא מוקלט, לא מחובר לקוד)

מסמך זה הוא שלב ההכנה בלבד לחיבור הקלטות אידיש להודעות המערכת של מודול
ה-`creditCard` (S3000–S3023). **אין בו שינוי קוד, אין העלאת קבצים, אין
שינוי DB/.env, אין פריסה.** בדומה ל-`ivr-audio-spec-v1.0-FROZEN.md` שקדם לו.

## 0. המבנה הרשמי — אומת

`WebFetch` ישיר ל-`pbx.tlivr.com` חוזר `403` (כנראה חוסם בוטים). המבנה
למטה אומת דרך פרוקסי קריאה (`r.jina.ai`) ב-3 קריאות עצמאיות עם ניסוחי
שאלה שונים (אחת בלי הטיה כלל, אחת עם רשימת מונחים לבדיקה, אחת שביקשה דמפ
מילולי מלא) — **כל השלוש התכנסו לאותו טקסט מילה-במילה**, כולל אותם ציטוטי
דוגמה, מה שנותן ביטחון גבוה שזה תוכן אמיתי ולא הזיה של שכבת הסיכום. בכל
זאת — זה לא זהה לפתיחת הדף בדפדפן שלך. אם תרצה ודאות מוחלטת, אתה עדיין
יכול להדביק כאן את הטקסט הרשמי להשוואה.

**ציטוט (Section: "systemMessages — החלפת הודעות מערכת לאורך השיחה"):**

> גם זה ערך כללי (לא `type` של מודול): מאפשר להחליף הודעות מערכת (קודי
> `S...` כמו הקראות מודול הסליקה, הודעות שגיאה וכו') בתוכן משלכם — לאורך
> כל המשך השיחה. כל קוד הודעת מערכת ממופה למערך קבצים זהה בדיוק למערך
> הקבצים הרגיל (`files`) — עם כל סוגי הערכים: `name` (קובץ), `text`
> (TTS), `SF` (הודעת מערכת אחרת), `number`, `digits`, `sum`, `fileLink`
> וכו'. ההחלפה תקפה מרגע שליחתה ועד סוף השיחה (או עד החלפה נוספת). שליחת
> קוד עם מערך ריק מחזירה אותו להודעת המערכת המקורית מה-DB.

```json
// פורמט 1 — מפה: קוד => מערך קבצים
"systemMessages": {
  "S3008": [{ "text": "כמה תשלומים תרצה לפרוס?" }],
  "S3013": [{ "name": "thanks_custom" }, { "SF": "S3014" }]
}

// פורמט 2 — מערך אובייקטים { sm, files }
"systemMessages": [
  { "sm": "S3008", "files": [{ "text": "כמה תשלומים?" }] }
]

// החזרה להודעת ברירת המחדל — מערך ריק
"systemMessages": { "S3008": [] }
```

```json
// דרך 1 — אלמנט עצמאי (חובה type:"systemMessages")
{ "type": "systemMessages",
  "systemMessages": { "S3008": [{ "text": "בחר/י מספר תשלומים" }] } }

// דרך 2 — מצורף ישירות למודול (נקבע *לפני* שהמודול רץ)
{ "type": "creditCard", "sum": 250, "payments": 3, "terminal": "42",
  "systemMessages": { "S3008": [{ "text": "בחר/י מספר תשלומים" }] } }
```

אלמנט עצמאי חייב `"type": "systemMessages"`. בלי `type` מוכר — המתקשר
נשלח חזרה לתפריט הקודם.

**שדות מערך `files` הכללי (מאותו תיעוד, אישור נוסף שזה אכן זהה למה שכבר
בשימוש בקוד שלנו):** `fileId`, `fileName` (קובץ קיים לפי שם), `text`
(TTS), `number` (מספר שלם מוקרא כמילים), `digits` (ספרה־ספרה),
`fileLink` (+ `fileName` לניהול cache), `activatedKeys`. **זה בדיוק הזוג
`{fileLink, fileName}` ש-`resolveAudioForProduction` כבר מחזיר היום
בפרודקשן** (`ivr-audio-resolver.service.js:49-53`) — אין צורך בשום שינוי
בפלט של ה-resolver עצמו.

### מסקנות שמתקנות את התוכנית המקורית (סעיפים 6–7 למטה עודכנו בהתאם)

1. **אין דרישת "כל 24" בכל בקשה.** המפה חלקית מטבעה — כוללים רק S-code
   שיש לו הקלטה מאושרת; לא כוללים כלל (או שולחים `[]`) לכל השאר, וטכנוליין
   כבר נופלת חזרה **מעצמה** להודעת ה-DB המקורית שלה. **לא צריך טקסט TTS
   עברי משלנו כ-fallback** להודעות האלה — זה מיותר, טכנוליין כבר עושה את
   זה.
2. **הוחלט (לא טעון עוד):** בשלב הראשון **לא** מוסיפים `sum`/`number`/
   `digits` למערכי `systemMessages` — כל S-code ממופה לקובץ שלו בלבד:
   `[{ "fileLink": "...", "fileName": "..." }]`. שומרים על ההתנהגות
   המקורית של מודול הסליקה (מה שהוא כבר עושה עם ההקראה הדינמית) ומחליפים
   רק את קובצי הודעות המערכת הסטטיות. **זו הנחה שדורשת אימות בפועל
   ב-trial** — לא ידוע מהתיעוד אם המנוע הפנימי ממשיך להקריא את הסכום/
   המספר אוטומטית אחרי fileLink חיצוני, או שזה היה עובד רק כי עד היום
   כל ה-S-codes תמיד ניגנו מברירת המחדל המובנית. יש להקשיב בפועל לשיחת
   trial ולוודא שהנתון הדינמי עדיין מושמע.
3. **הוחלט (לא טעון עוד):** בשלב הראשון **לא** מוסיפים `SF` בין הודעות —
   חשש מהשמעה כפולה אם מנוע הסליקה כבר ממשיך בעצמו להודעה הבאה. זוגות
   S3009↔S3010 ו-S3013↔S3014 כל אחד מקבל קובץ נפרד משלו, בלי שרשור.
4. **המיקום הטבעי בקוד הוא צירוף `systemMessages` ישירות לתוך האובייקט
   שמחזיר `creditCardModule()`** (ivr.js:276) — בדיוק כמו הדוגמה הרשמית
   `{"type":"creditCard",...,"systemMessages":{...}}` — לא כמודול נפרד.

### עדכון קריטי — מצב `IVR_AUDIO_MODE` בפרודקשן

לפי המידע שקיבלתי ממך עכשיו, `IVR_AUDIO_MODE` **כבר `on`** בפרודקשן
לתפריטים הרגילים (לא `off` כפי שהיה בזמן הכתיבה המקורית). **המשמעות:**
אסור להשתמש שוב ב-`IVR_AUDIO_MODE`/`IVR_AUDIO_TRIAL_CALLER_PHONE` עבור
ה-`systemMessages` — זה יפעיל אותם לכולם באופן מיידי. `ivr-audio-mode.
service.js` (`parseAudioMode`/`shouldUseAudioForCall`) **גנרי לחלוטין**
— לא תלוי בשם משתנה סביבה ספציפי, השיוך לשם המשתנה קורה רק בנקודת
הקריאה. ההצעה (**טעונה אישורך, טרם מומשה**): משתנה סביבה **נפרד**, למשל
`IVR_AUDIO_PAYMSG_MODE` (ברירת מחדל `off`), הנקרא רק בנקודה שמצרפת
`systemMessages` ל-`creditCardModule()`, תוך שימוש חוזר באותן שתי
פונקציות קיימות ללא שינוי בהן. כך ה-trial של הסליקה לגמרי עצמאי מהמצב
הפעיל היום של התפריטים הרגילים.

## 1. ממצאי בדיקת הקוד הקיים

`creditCardModule(amount)` ב-[ivr.js:276-289](../../frontend/backend/ivr.js#L276-L289)
הוא המקום היחיד שבונה את תשובת ה-`creditCard`:

```js
function creditCardModule(amount) {
  return {
    type: "creditCard",
    name: "payment",
    sum: amount,
    sumChangeable: "no",
    cvv: "yes",
    tz: "yes",
    payments: 1,
    category: "תרומה",
    terminal: process.env.CREDIT_CARD_TERMINAL || "",
  };
}
```

נקראת מ-4 מקומות ([ivr.js:558,568,596,607](../../frontend/backend/ivr.js#L558)),
תמיד אחרי `if (!hasTerminal) return paymentPlaceholder(audio);` — כלומר כבר
יש שער קיים שמונע קריאה כשאין טרמינל מוגדר.

**כרגע אין שום מפתח `systemMessages` בתשובה.** זה בדיוק המקום שבו יתווסף
המפתח, ברגע שיאומת המבנה הרשמי (סעיף 0). שום שדה קיים אחר (`sum`,
`sumChangeable`, `cvv`, `tz`, `payments`, `category`, `terminal`) לא ישתנה.

`audio` (הקשר ה-resolver) כבר מגיע כפרמטר לפונקציית האם שקוראת ל-
`creditCardModule` בכל 4 המקומות (`buildResponse`), כך שאין צורך להעביר
אותו הלאה בשרשור נוסף — `creditCardModule` יכול לקבל אותו כפרמטר נוסף
כשיגיע הזמן לממש.

`CREDIT_CARD_TERMINAL` עדיין ריק כברירת מחדל (`|| ""`) — פריט נפרד, לא
קשור להקלטות, שכבר תועד כפתוח מול טכנוליין ב-FROZEN v1.0 סעיף 7.

## 2. מוסכמת שמות — הבחנה בין S-code לבין audioId פנימי

| | דוגמה | תפקיד |
|---|---|---|
| **S-code** | `S3000` | המפתח שטכנוליין מגדירה בפרוטוקול שלה (חיצוני, קבוע, לא בשליטתנו) |
| **audioId פנימי** | `PAYMSG-3000` | השורה שלנו ב-`ivr_audio_recordings` (DB), נגזרות קבצים, resolver |

קידומת **`PAYMSG-`** נבחרה במיוחד כדי לא להתנגש עם `PAY-001..008` הקיים
(שמתייחס להודעות התפריט הפנימי *לפני* הכניסה למודול הסליקה — "הזינו סכום",
"הסכום גבוה מדי" וכו', שנשארות ללא שינוי). המספר אחרי המקף זהה למספר
ה-S-code (3000–3023) בכוונה, כדי שהמיפוי S3000↔PAYMSG-3000 יהיה טריוויאלי
לעין ולבדיקה, בלי צורך בטבלת תרגום נפרדת בעתיד.

קטגוריה חדשה ב-DB: `category: "paymsg"` (מקבילה ל-`open`/`menu`/`debt`/...
הקיימות).

שם קובץ מומלץ להעלאה (תואם למוסכמה הישנה ב-FROZEN v1.0 סעיף 4):
`ivr_paymsg_<Sxxxx>_yi_v1.wav` — למשל `ivr_paymsg_s3000_yi_v1.wav`.
הנגזרת (PCM 8kHz) תיווצר אוטומטית ע"י `convert-ivr-audio-to-wav.js` הקיים,
ללא שינוי במנגנון.

| S-code | הנתון הדינמי שבא אחריו |
|---|---|
| S3000 | סכום התשלום (שקלים) |
| S3009 | מספר התשלומים המקסימלי האפשרי |
| S3012 | טקסט/קוד הודעת השגיאה מהסולק |
| S3014 | מספר האישור בפועל |

S3010 ("תשלומים") הוא מקרה מיוחד — הוא **בא אחרי** הנתון הדינמי של S3009,

S3013↔S3014 הם ככל הנראה זוגות דומים (הודעה עצמאית + זנב עם נתון), אך
בלי לראות את התיעוד הרשמי אי אפשר לאשר אם S3011/S3013 תמיד מנוגנות בפני
עצמן או רק כשאין המשך — **זה בדיוק סוג הפרט שהתיעוד הרשמי (סעיף 0) יבהיר.**

S3002/S3021/S3022 — אלו *מבקשות* קלט, לא *מקריאות* נתון, ולכן אינן נחשבות
"חיתוך לפני דינמי" למרות שמגיע אחריהן קלט מהמשתמש).

## 6. תוכנית חיבור (סעיף 8 בבקשה) — טרם מבוצע

כל השלבים משתמשים במנגנון **הקיים** ללא שינוי לוגיקה, רק בהיקף נתונים חדש:

1. **מיפוי הזרעה (seed)** — קובץ חדש בסגנון `seed-ident-audio.js` הקיים
   (`seed-paymsg-audio.js`?), עם 24 השורות (`audioId: PAYMSG-3000..3023`,
   `category: "paymsg"`, `sourceTextHe`) — **רק אחרי** שהעברית/מבנה יאושרו
   כאן. אין קשר ל-`IVR_AUDIO_CANONICAL_RECORDINGS`/`IDENT_RECORDINGS`
   הקיימים — קובץ שלישי נפרד, מיובא באותו האופן שבו `ivr-audio-context.
   service.js` כבר מייבא את `IDENT_RECORDINGS` (התיקון האחרון בדיוק לימד
   אותנו לא לשכוח לאחד מקורות טקסט/מיפוי).
2. **העלאה** — 24 קבצי WAV/מקור דרך מנגנון ההעלאה הקיים לאדמין
   (`audioFile1`), אחרי הקלטה בפועל.
3. **המרה לנגזרת** — `convert-ivr-audio-to-wav.js` הקיים, ללא שינוי
   (`computeDerivedFilename` כבר גנרי לכל audioId).
4. **אישור** — זרימת האישור הקיימת באדמין (`status: draft → approved`),
   ללא שינוי מנגנון.
5. **בניית fileLink** — `resolveAudioForProduction(audioId, fallbackText)`
   הקיים כבר עובד לכל audioId שיש לו שורת DB תקינה — **אין צורך בשינוי
   ב-resolver עצמו**, רק בקוד שקורא לו (סעיף הבא).
6. **שילוב ב-`systemMessages`** *(מעודכן לפי החלטותיך)* — קוד חדש
   ב-`creditCardModule()` (ivr.js:276) שמצרף מפתח `systemMessages` **חלקי**
   (רק S-codes שיש להם הקלטה מאושרת — לא צריך "מלא 24"), ע"י מיפוי
   S300X→PAYMSG-300X, קריאה ל-`resolveAudio` לכל אחד, והשמטת המפתח
   כליל עבור כל S-code שנכשל (**לא** צריך טקסט TTS עברי כ-fallback —
   טכנוליין כבר נופלת חזרה לבד להודעת ה-DB המקורית שלה כשלא כוללים
   מפתח). **שלב ראשון: קובץ יחיד בלבד לכל S-code** —
   `"S300X": [{ "fileLink": "...", "fileName": "..." }]` — בלי `sum`/
   `number`/`digits`, בלי `SF`. שמירה על התנהגות מודול הסליקה המקורית
   פרט להחלפת קובץ ההודעה עצמו.
7. **Trial — מנגנון נפרד, לא `IVR_AUDIO_MODE` הקיים** *(מעודכן: קיבלתי
   ממך שהמצב כבר `on` בפרודקשן לתפריטים הרגילים)* — הצעה: משתנה סביבה
   חדש, למשל `IVR_AUDIO_PAYMSG_MODE` (`off`/`trial`/`on`, ברירת מחדל
   `off`), הנקרא **רק** בנקודה שמצרפת `systemMessages` ל-
   `creditCardModule()`, תוך שימוש חוזר ב-`parseAudioMode`/
   `shouldUseAudioForCall` הקיימים כמו שהם (הם כבר גנריים לגמרי — לא
   קשורים לשם המשתנה). כך התפריטים הרגילים (כבר `on`) לא נוגעים בכלל,
   וה-systemMessages מתחיל תמיד ב-`off` משלו, עצמאי לגמרי.
8. **הפעלה מלאה** — מעבר `IVR_AUDIO_PAYMSG_MODE=on` ב-.env בפרודקשן
   (פעולה ידנית שלך, לא שלי) אחרי שה-trial של הסליקה אושר בנפרד,
   ובלי לגעת ב-`IVR_AUDIO_MODE` הקיים בכלל.

### 8.2 קבצים שישתנו (רשימה סופית, טרם בוצע)

1. **חדש:** `frontend/backend/seed-paymsg-audio.js` — מראה מדויקת של

   היום — לא רץ אוטומטית ב-startup). `translation` נשאר ריק (אין תרגום
   סופי כרגע, כמו שביקשת).
2. **`frontend/settings.html`** — שורה אחת נוספת: כפתור שלישי
   `<button class="ivr-sheet-tab-btn" data-sheet="paymsg">💳 הודעות סליקה</button>`
   ליד שני הכפתורים הקיימים (היום בשורות 88-91). שום שינוי במבנה הטבלה
   עצמה (8 העמודות נשארות זהות לכל הגיליונות).
3. **`frontend/js/settings.js`** — 3 מקומות, כולם שינוי קטן ומקומי:
   - `matchesSheet()` (שורה 1502-1505) — היום בדיקה בינארית
     (`sentences`/הכל-האחר) לפי `SHEET1_CATEGORIES`. יהפוך לבדיקה
     תלת-כיוונית: `sentences` → `SHEET1_CATEGORIES`, `paymsg` →
     `category === "paymsg"`, `numbers` → כל השאר.
   - `addRow()` (שורה 1922) — היום `activeSheet==="sentences"?"open":"number"`.
     יהפוך למיפוי תלת-כיווני (כולל `paymsg` → `"paymsg"`) כדי ששורה חדשה
     שנוספת בזמן שהגיליון השלישי פתוח תתויג נכון מיד.
   - `exportToExcel()` (שורה 1949-1955) — הוספת `sheet3Rows` (סינון
     `category==="paymsg"`) ו-`XLSX.utils.book_append_sheet(wb,...,"גיליון3")`.
   - **בדקתי ואישרתי:** מנגנון לחיצת הטאבים עצמו (שורה 1998-2008) כבר
     גנרי לגמרי — קורא `data-sheet` מהכפתור שנלחץ בפועל — **כפתור שלישי
     "עובד" בלי לגעת בקוד הזה בכלל.**
4. **ללא שינוי בכלל:** `server.js`, `db.js` (כל ה-CRUD, ה-multer upload
   route, ה-DB schema), `convert-ivr-audio-to-wav.js` — כולם כבר גנריים
   לחלוטין לפי `audioId`/`category`, לא יודעים ולא צריכים לדעת ש-PAYMSG
   קיים.

### 8.3 איך העלאה/החלפה עובדת (מנגנון קיים, ללא שינוי)

`POST /api/admin/ivr-audio/:id/audio/:slot` (`server.js:577`) — `id`
= `PAYMSG-3000` וכו', `slot`=1 תמיד (אין צורך ב-2/3 — כל S-code צריך
קובץ אחד). קובץ חדש לאותה שורה **מוחק אוטומטית את הקובץ הקודם** מהדיסק
(`server.js:588-591`) ומעדכן את `audioFile1` ב-DB — זו כבר "החלפה", לא
צריך מנגנון נוסף.

### 8.4 גרסה/cache — כבר פתור אוטומטית, ללא קוד חדש

כל העלאה (גם לאותו S-code) מקבלת שם קובץ עם **סיומת אקראית טרייה בת 16
תווים** (`crypto.randomBytes(8)`, `server.js:564`) — כלומר `fileName`
משתנה בכל העלאה מעצמו, גם אם `audioId` זהה. זה בדיוק מה שביקשת בסעיף 7 —
**קיים כבר לכל 83 ההקלטות הנוכחיות, לא ייחודי ל-PAYMSG.**

הערה תפעולית (לא קוד): כדי שהקובץ החדש יהיה שמיש בפועל דרך `fileLink`
צריך גם להריץ את סקריפט ההמרה `convert-ivr-audio-to-wav.js --apply`
(ידני, כמו שנעשה היום ל-83 ההקלטות הקיימות) אחרי אישור השורה — זה תהליך
תפעולי קיים, לא שינוי קוד, ולא ייחודי ל-PAYMSG. אם תרצה שזה יקרה
אוטומטית בעתיד, זו החלטה נפרדת שלא נכללת כאן.

### 8.5 איך ה-IVR יקבל את הקובץ החדש (תכנון עתידי, טרם מומש — שלב נפרד)

זה **לא** חלק מהעבודה הנוכחית (רק ממשק הניהול) — אבל כדי לענות על סעיף
15: ברגע שתאשר שלב נפרד, `creditCardModule()` (`ivr.js:276`) יקרא
ל-`resolveAudioForProduction("PAYMSG-300X", ...)` הקיים (גנרי, ללא
שינוי) לכל S-code, ויצרף רק את מי שהצליח ל-`systemMessages` בתור
`{ "S300X": [{ "fileLink": "...", "fileName": "..." }] }` — בדיוק
כפי שאושר. הכל תחת `IVR_AUDIO_PAYMSG_MODE` (off/trial/on, ברירת מחדל
off) — נפרד לגמרי מ-`IVR_AUDIO_MODE` הקיים, לפי סעיף 14 בבקשתך.

### 8.6 חזרה בטוחה לברירת המחדל של טכנוליין

לכל S-code בלי שורה מאושרת (`status !== "אושר"`) או בלי קובץ נגזרת תקין —
המפתח **לא ייכלל בכלל** ב-`systemMessages` (לא ריק, לא TTS משלנו). לפי
המבנה הרשמי שאומת: S-code שלא מופיע במפה = טכנוליין מנגנת את הודעת ברירת
המחדל שלה בעצמה. זה חל שורה-שורה — אפשר להעלות רק חלק מ-24 ולהתקדם
בהדרגה.

### 8.7 מה עוד לא הוחלט (טעון תשובתך לפני מימוש)

- ה"הערה מה להקליט" לכל שורה (עמודת `הסבר שימוש`) — האם להזריע עם ניסוח
  קצר שלי (כמו "משפט נחתך לפני הסכום") או שתרצה לכתוב את זה בעצמך אחרי
  שהגיליון קיים?
- שם התווית על הכפתור השלישי — הצעתי "💳 הודעות סליקה", אפשר לשנות.

## 9. תוכנית מתוקנת וסופית (גרסה 2 — כוללת המרה אוטומטית) — טרם מומשה

תיקון חובה שקיבלתי: **אסור** לדרוש כניסה לשרת/הרצת
`convert-ivr-audio-to-wav.js --apply` ידנית אחרי כל העלאה. כל התהליך —
בדיקת פורמט, המרה, גרסת cache — חייב לקרות אוטומטית דרך האתר. הפרקים
הבאים מחליפים את מה שנכתב על כך בסעיף 8.

### 9.1 ממצא מפתח שמאפשר את זה בלי סיכון

`scripts/convert-ivr-audio-to-wav.js` כבר מייצא (`module.exports`, שורות
238-247) את **כל הלוגיקה הטהורה** שצריך: `computeDerivedFilename`,
`computeTmpFilename`, `isPathContained`, `isReadyAsIs`,
`isValidDerivedProbe`, `decideAction`, `buildFfprobeArgs`,
`buildFfmpegArgs` — כולן כבר מכוסות בבדיקות קיימות
(`convert-ivr-audio-to-wav.test.js`). **התוכנית היא להשתמש באותן הפונקציות
המדויקות בתוך ה-route של ההעלאה, לא לכתוב לוגיקת המרה חדשה.** רק שכבת
ה-I/O (`probeAudio`/`convertToTmpWav`, שקוראות בפועל ל-`ffprobe`/`ffmpeg`
דרך `execFileSync`) עוברת משימוש בסקריפט CLI נפרד לשימוש ישיר בתוך
ה-handler של ה-route — אותו קוד בדיוק, מקום הפעלה שונה.

### 9.2 הזרימה החדשה של `POST /api/admin/ivr-audio/:id/audio/:slot`

המצב **היום** (`server.js:577-606`): multer שומר את הקובץ החדש בדיסק →
**מייד מוחקים את הקובץ הישן** (שורה 588-591) → מעדכנים DB. זה בדיוק מה
שאסור (סעיף 8 הישן, ונקודה 8 בבקשתך האחרונה).

**הזרימה החדשה המתוכננת:**

1. multer שומר את הקובץ החדש בדיסק (שם עם סיומת אקראית טרייה — קיים,
   ללא שינוי) — הקובץ **הישן עדיין קיים ופעיל**, לא נגעו בו.
2. `probeAudioSafe(newFilePath)` (מיובא מהסקריפט) — אם נכשל (לא קובץ
   שמע תקין בכלל) → כישלון, עוברים לשלב 5.
3. `decideAction`/`isReadyAsIs` (מיובאים, אותה לוגיקה בדיוק כמו הסקריפט):
   - אם הקובץ כבר PCM s16le/8000/mono **וגם** סיומתו `.wav` — מוכן כמו
     שהוא, אין צורך בהמרה, ממשיכים לשלב 4.
   - אחרת — `convertToTmpWav()` (קורא בפועל ל-`ffmpeg`, אותה פונקציה
     בדיוק מהסקריפט) ליצור קובץ נגזר זמני, `probeAudioSafe` על התוצאה,
     `isValidDerivedProbe` לאימות. נכשל בכל שלב → כישלון, שלב 5.
     הצליח → `fs.renameSync` אטומי מהזמני לשם הנגזרת הסופי (בדיוק כמו
     שהסקריפט עושה היום, שורה 212 בסקריפט) — **שם הנגזרת כבר כולל את
     הסיומת האקראית הטרייה של הקובץ החדש, אז זו גרסה חדשה אוטומטית,
     בלי קוד נוסף (סעיף 9.4).**
4. **הצלחה:** רק עכשיו — מוחקים את הקובץ/הנגזרת **הישנים** (אם קיימים
   ושונים מהחדשים), מעדכנים `audioFile1` + סטטוס ב-DB (כמו היום), רושמים
   audit log, מחזירים `200 OK` לדפדפן.
5. **כישלון (בכל שלב):** מוחקים רק את **הקובץ החדש** שהועלה כרגע (ואת
   קובץ ה-`.tmp` אם נוצר) — **לא נוגעים בקובץ/בנגזרת/ב-DB של ההקלטה
   הפעילה הקיימת בכלל**. מחזירים `400`/`422` עם הודעת שגיאה בעברית ברורה
   (למשל: *"הקובץ שהועלה אינו קובץ שמע תקין, או שההמרה נכשלה. ההקלטה
   הקודמת נשארה פעילה. נסו קובץ אחר."*). הדפדפן מציג את זה ומאפשר ניסיון
   חוזר מיד (אותו כפתור בחירת קובץ, אין ריענון דף נדרש).

זה בדיוק עונה על נקודות 6, 7, 8 בבקשתך: בדיקת פורמט אוטומטית, המרה
אוטומטית כשצריך, גרסה חדשה תמיד, ואפס פגיעה בהקלטה הפעילה לפני שהחדשה
אומתה בהצלחה.

### 9.3 קבצים שישתנו (מעודכן)

בנוסף לרשימה בסעיף 8.2 (`seed-paymsg-audio.js` חדש, `settings.html`
שורה אחת, `settings.js` 3 מקומות קטנים) — **שינוי אחד נוסף, מהותי**:

- **`server.js`** — ה-handler של `POST /api/admin/ivr-audio/:id/audio/:slot`
  (שורות 577-606) נכתב מחדש לפי הזרימה בסעיף 9.2: מוסיף `require` ל-4
  הפונקציות הטהורות מ-`scripts/convert-ivr-audio-to-wav.js`, מזיז את
  מחיקת הקובץ הישן להיות **אחרי** אימות הצלחה (לא לפני), ומוסיף את שלב
  הבדיקה/ההמרה בין שמירת הקובץ החדש לעדכון ה-DB. שום שינוי בחוזה ה-API
  כלפי חוץ מלבד: תשובת שגיאה חדשה אפשרית (`400`/`422` עם טקסט עברי) אם
  ההמרה נכשלה — שלא הייתה קיימת קודם (קודם כל העלאה "הצליחה" מיידית).
- **`ivr.js`** — (שלב נפרד, סעיף 9.5 למטה) `creditCardModule()` מקבל 2
  פרמטרים נוספים (`audio`, `callerPhone`), ובונה `systemMessages`.
- **חדש:** `frontend/backend/ivr-audio-paymsg-conversion.service.js`
  (הצעה) — עוטף את 4 הפונקציות הטהורות + את שכבת ה-I/O
  (`probeAudio`/`convertToTmpWav`) לפונקציה אחת `convertUploadedFile
  (rawPath, uploadDir) -> {ok, derivedFilename} | {ok:false, reason}` —
  כדי שה-route ב-`server.js` יישאר קצר, וכדי שאפשר לבדוק את הזרימה
  המלאה (הצלחה/כישלון/ניקוי) בלי להריץ Express אמיתי.
- **ללא שינוי:** `db.js`, ה-DB schema, שאר ה-routes.

### 9.4 גרסה/cache — עדיין אוטומטי, ללא עמודת DB חדשה

שם הקובץ הגולמי כבר כולל סיומת אקראית טרייה (multer, `server.js:564`,
ללא שינוי). שם הנגזרת (`computeDerivedFilename`) נגזר תמיד מהשם הגולמי —
כך שכל העלאה מקבלת **גם קובץ גולמי חדש וגם קובץ נגזרת חדש**, אוטומטית,
בלי צורך בעמודת "גרסה" נפרדת ב-DB. ה-resolver הקיים
(`ivr-audio-resolver.service.js`) כבר מחשב את שם הנגזרת מחדש בכל קריאה —
**אפס שינוי נדרש ב-resolver עצמו.**

### 9.5 חיבור בפועל ל-`creditCardModule` (סעיף 10 בבקשתך)

`creditCardModule(amount)` (`ivr.js:276`) יהפוך ל-
`creditCardModule(amount, audio, callerPhone)` (שני פרמטרים נוספים,
כבר זמינים בכל 4 מקומות הקריאה דרך `buildResponse(query, donor, audio)`
— `audio` כבר בסקופ, `callerPhone` הוא `phone` המנורמל שכבר מחושב
ב-`ivr.service.js` ויועבר הלאה כפרמטר חדש ל-`buildResponse` עצמו).

בתוך `creditCardModule`, אם `shouldUseAudioForCall({mode:
process.env.IVR_AUDIO_PAYMSG_MODE, phone: callerPhone, trialPhone:
process.env.IVR_AUDIO_PAYMSG_TRIAL_CALLER_PHONE})` (מ-`ivr-audio-mode.
service.js` הקיים, ללא שינוי — גנרי לגמרי) מחזיר `true`:

```js
var PAYMSG_S_CODE = { "PAYMSG-3000":"S3000", "PAYMSG-3001":"S3001", /* ... */ "PAYMSG-3023":"S3023" };
function buildPaymsgSystemMessages(audio) {
  var out = {};
  Object.keys(PAYMSG_S_CODE).forEach(function (audioId) {
    var result = audio.resolveAudioId(audioId, ""); // "" — בכוונה, ראה 9.6
    if (result && result.fileLink) {
      out[PAYMSG_S_CODE[audioId]] = [{ fileLink: result.fileLink, fileName: result.fileName }];
    }
  });
  return out;
}
```

ואז מצרפים `systemMessages: buildPaymsgSystemMessages(audio)` לאובייקט
המוחזר **רק אם יש לפחות מפתח אחד** (אין טעם לשלוח `systemMessages: {}`
ריק). **בלי `SF`, בלי `sum`/`number`/`digits`, בלי שינוי סדר** — בדיוק
כמו שביקשת בנקודה 10-11. כשה-mode הוא `off` (ברירת המחדל) — המפתח לא
נבנה בכלל, ההתנהגות זהה ל-100% למה שיש היום.

### 9.6 חזרה בטוחה לברירת המחדל (סעיף 12)

`audio.resolveAudioId("PAYMSG-300X", "")` — פס ריק בתור fallbackText
בכוונה (לא טקסט TTS עברי): אם השורה לא `"אושר"` או שאין קובץ תקין, ה-
resolver הקיים כבר מחזיר `{text:""}` או טקסט גנרי — אבל `buildPaymsg
SystemMessages` **בודק במפורש `result.fileLink`** ומדלג לגמרי אם אין
כזה, כך שאף פעם לא נכנס פריט `{text:...}` לתוך `systemMessages` — בהתאם
למבנה הרשמי שאומת: קוד שלא מופיע במפה = טכנוליין מנגנת ברירת מחדל שלה.

### 9.7 מנגנון ה-trial — בדיוק מי נכנס (סעיף 11)

`IVR_AUDIO_PAYMSG_MODE` **נפרד** מ-`IVR_AUDIO_MODE` (לא נוגעים בקיים).
מוצע גם משתנה נפרד `IVR_AUDIO_PAYMSG_TRIAL_CALLER_PHONE` (נפרד מ-
`IVR_AUDIO_TRIAL_CALLER_PHONE` הקיים) — כדי שתוכל לבחור מספר בדיקה
שונה (או זהה, לבחירתך) לזה ששימש לניסוי התפריטים הרגילים, בלי שהשניים
משפיעים זה על זה.

- `off` (ברירת מחדל): `systemMessages` לעולם לא נבנה. זהה להיום.
- `trial`: נבנה **רק** כש-`callerPhone` המנורמל (מ-`PBXphone`) שווה
  בדיוק ל-`IVR_AUDIO_PAYMSG_TRIAL_CALLER_PHONE` המנורמל. כל מתקשר אחר —
  זהה ל-`off`. זו **אותה פונקציה גנרית בדיוק** (`shouldUseAudioForCall`)
  שכבר עובדת היום ל-`IVR_AUDIO_MODE`, לא קוד חדש.
- `on`: נבנה לכל שיחה, לכל מתקשר.

**מגבלה טכנית קיימת ומוכחת (לא חדשה, לא ייחודית ל-PAYMSG — סעיף 13):**
שינוי `IVR_AUDIO_PAYMSG_MODE`/`IVR_AUDIO_PAYMSG_TRIAL_CALLER_PHONE`
עצמם ב-`.env` **דורש `pm2 restart --update-env`** כדי להיכנס לתוקף —
זה נכון היום גם ל-`IVR_AUDIO_MODE` הקיים (משתני סביבה נטענים פעם אחת
בהפעלת התהליך). **זה לא חל על החלפת הקלטה** — זו נקראת מחדש מה-DB/דיסק
בכל שיחה, בלי שום restart (סעיף 9.8). בקיצור: להחליף איזה S-code
מוקלט → אין restart בכלל; להדליק/לכבות את כל תכונת ה-trial/on עצמה →
כן restart, בדיוק כמו התכונה הקיימת.

### 9.8 למה החלפת הקלטה נכנסת לשימוש אוטומטית (סעיף 13)

`resolveAudioForProduction`/`getRecordByAudioId`/`fs.existsSync` —
כולם נקראים **מחדש בכל שיחה**, בלי שום cache בזיכרון (אומתי בקוד
הקיים, `ivr-audio-resolver.service.js`). ברגע שה-DB מעודכן (סעיף 9.2
שלב 4) והקובץ החדש על הדיסק — השיחה **הבאה** (לא הנוכחית, אם היא כבר
באמצע) כבר תקבל את הקובץ החדש. אין קוד, אין CSV, אין סקריפט, אין
restart — בדיוק כמו שביקשת.

### 9.9 תוכנית בדיקות מלאה (סעיף 16)

בהמשך למוסכמה הקיימת (Node פשוט, בלי מסגרת בדיקה, DB/קבצים זמניים
אמיתיים במקום מוקים):

1. **`seed-paymsg-audio.js`** — 24 שורות בדיוק, `audioId`/`category`
   נכונים, אין כפילות, ריצה כפולה אידמפוטנטית (לא דורסת שורה קיימת).
2. **המרה אוטומטית — הצלחה:** קובץ MP3/לא-WAV תקין → מועלה → נוצרת
   נגזרת PCM8k תקינה בשם חדש → DB מתעדכן → תשובת `200`.
3. **המרה אוטומטית — כבר תקין:** קובץ WAV שכבר PCM s16le/8000/mono →
   לא רץ ffmpeg בכלל (`isReadyAsIs`), משתמשים בו ישירות — כמו שהסקריפט
   הקיים עושה.
4. **כישלון — קובץ לא תקין:** טקסט/תמונה בשם `.wav` → `probeAudioSafe`
   נכשל → הקלטה קיימת (אם יש) **לא נמחקת**, DB **לא** מתעדכן, קובץ
   שהועלה עכשיו נמחק, תשובת `400` עם הודעה בעברית.
5. **כישלון — ffmpeg נכשל בפועל** (למשל קובץ אודיו פגום בתוכן): אותה
   התנהגות כמו #4 — בדיקה עם mock/stub ל-`execFileSync` שזורק.
6. **סדר פעולות בטוח:** בדיקה מפורשת שבמהלך הצלחה, מחיקת הקובץ הישן
   קורית **אחרי** ש-DB עודכן בהצלחה (לא לפני) — ליבת הדרישה של נקודה 8.
7. **`buildPaymsgSystemMessages`:** DB זמני אמיתי (לא מוק) עם שילוב
   שורות `"אושר"`/`"חסר"`/סטטוסים אחרים — מוודא שרק שורות מאושרות עם
   קובץ תקין נכנסות למפה, בדיוק בפורמט `[{fileLink,fileName}]`, בלי
   `SF`/`sum`/`number`/`digits` בשום מקרה.
8. **Trial scoping:** `IVR_AUDIO_PAYMSG_MODE=trial` + טלפון תואם/לא
   תואם → `systemMessages` מופיע/לא מופיע בהתאם; ומוודא ש-
   `IVR_AUDIO_MODE` (התפריטים הרגילים) לא מושפע כלל משינויים ב-
   `IVR_AUDIO_PAYMSG_MODE` ולהפך.
9. **רגרסיה מלאה** — כל הבדיקות הקיימות (185+ נכון להיום) ממשיכות
   לעבור ללא שינוי כש-`IVR_AUDIO_PAYMSG_MODE` לא מוגדר (off).
10. **UI (ידני, כמו כל שאר settings.html היום — אין מסגרת בדיקות
    frontend בפרויקט):** רשימת QA — הגיליון השלישי מציג 24 שורות; העלאת
    קובץ תקין מצליחה ומשמיעה נגן מעודכן; העלאת קובץ פגום מציגה שגיאה
    בעברית וההקלטה הקודמת עדיין מתנגנת בנגן; ייצוא Excel כולל גיליון3.

### 9.10 תוכנית Rollback (סעיף 16, מעודכן — ראה גם 9.11 להיסטוריית גרסה)

- **הקלטה בודדת גרועה שכבר אושרה:** להעלות קובץ מתוקן דרך אותה שורה
  (אותה זרימה בטוחה, סעיף 9.2) — נכנס לשימוש בשיחה הבאה, בלי קוד. או:
  לשנות את הסטטוס בחזרה (למשל ל-"נבדק") דרך התפריט הקיים — מפסיק מיידית
  את השימוש בה, נופל לברירת המחדל של טכנוליין, גם בלי restart. **או:**
  שחזור הגרסה הקודמת בלחיצה אחת (סעיף 9.11) — בלי להעלות שום קובץ מחדש.
- **לכבות את כל תכונת ה-PAYMSG בפרודקשן:** אם `IVR_AUDIO_PAYMSG_MODE`
  מעולם לא הוגדר (או `off`) — אין מה לבטל, זו ברירת המחדל. אם הודלק
  (`trial`/`on`) וצריך לכבות מהר — `IVR_AUDIO_PAYMSG_MODE=off` ב-.env +
  `pm2 restart --update-env` (אותה פעולה בדיוק כמו הדלקה, לא פעולה
  הרסנית, לא נוגעת ב-`IVR_AUDIO_MODE`).
- **Rollback ברמת קוד** (למשל באג בלוגיקת ההמרה עצמה): `git revert`
  לקומיט הרלוונטי + פריסה רגילה. אין שינוי סכמת DB בכל התוכנית הזו —
  שורות `PAYMSG-*` שנשארות ב-`ivr_audio_recordings` אחרי revert הן
  שורות "יתומות" זמינות אך לא-מקושרות-לקוד, לא מזיקות (בדיוק כמו שהיה
  קורה עם `IDENT-*` אילו הוחזר `ivr.js` לאחור).

### 9.11 היסטוריית גרסה + Rollback אמיתי — הפתרון המוצע (טעון אישור, בלי סכמה חדשה)

**הבהרה שהתקבלה:** גם *אחרי* החלפה מוצלחת, לא מספיק "להעלות שוב את
הקובץ הישן" כרולבק — צריך מנגנון ברור. וגם: לא למחוק את הישן כלל עד
שהחדש אומת (כבר בתוכנית, 9.2), אבל עכשיו גם *לשמר* אותו זמנית אחרי
ההחלפה, לא רק לפניה.

**הפתרון המוצע — אפס שינוי סכמה, שימוש חוזר במבנה קיים:** לכל שורה כבר
יש **3 סלוטים עצמאיים** (`audioFile1/2/3`) — היום שלושתם קיימים אבל
שום קוד לא נותן להם משמעות (לא ה-resolver, שקורא רק `audioFile1`).
**ההצעה: `audioFile2` הופך למשבצת "הגרסה הקודמת" האוטומטית, רק לשורות
`category="paymsg"`** (לא נוגעים בהתנהגות עבור 83 השורות הקיימות):

בהעלאה מוצלחת לסלוט 1 של שורת PAYMSG:
1. (כמו ב-9.2) הקובץ החדש כבר אומת/הומר בהצלחה, עדיין לא ב-DB.
2. `oldFile2` (הגרסה-לפני-הקודמת, אם יש) — **נמחקת מהדיסק עכשיו** (זו
   הפעם היחידה שמוחקים קובץ במחזור הזה — קובץ שכבר "יצא" מחלון הרולבק).
3. עדכון DB **אחד**: `audioFile2 = audioFile1 הישן` (הופך לגיבוי),
   `audioFile1 = הקובץ החדש` (הופך לפעיל). **הקובץ הפיזי הישן שהיה
   ב-audioFile1 לא נמחק בכלל** — הוא עכשיו מוגן תחת `audioFile2`.
4. תוצאה: תמיד יש בדיוק "גרסה פעילה" (סלוט 1) ו"גרסה קודמת אחת" (סלוט
   2) — לא היסטוריה שלמה, אלא רולבק-צעד-אחד, שזה מספיק כדי לחזור אחורה
   מיד בלי לאבד את מה שהיה פעיל.

**שחזור (Rollback UI):** route חדש קטן, `POST /api/admin/ivr-audio/:id/
restore-previous` — **מוחלט מדובר בהחלפת ערכים בין שתי עמודות טקסט
קיימות ב-DB, בלי לגעת בשום קובץ בדיסק בכלל** (שני הקבצים כבר שם, רק
מתחלפת ההפניה): `audioFile1 ↔ audioFile2`. מיידי, בלי restart, בלי
להעלות שום דבר. הסטטוס עצמו לא משתנה (הוא ממילא לא-יורד-אוטומטית לפי
המנגנון הקיים) — כך שאם השורה הייתה "אושר" לפני ההחלפה הגרועה, היא
נשארת "אושר" אחרי השחזור, והגרסה הטובה חוזרת לשימוש מיידית.

**זה בהחלט תוסף חדש (לא רק תיעוד)** — route חדש אחד + שינוי קטן בזרימת
ההעלאה (סעיף 9.2 שלב 4, כדי לבצע את ה-shift ל-`audioFile2` במקום
פשוט למחוק) — **בלי שום `CREATE TABLE`/`ALTER TABLE`/עמודה חדשה.**
מוגבל ל-`category="paymsg"` כברירת מחדל בהצעה הזו (לא משנה התנהגות
upload עבור 83 השורות הקיימות, שלא ישתמשו ב-`audioFile2` בתור גיבוי
אוטומטי אלא אם תבקש להרחיב את זה גם אליהן — נפרד, לא כלול כאן).

### 9.12 תיקון קריטי — הפרדת "פעיל" מ-"ממתין לאישור" (מחליף חלק מ-9.11)

**האבחון שלך מדויק, עם תיקון טכני קטן אחד:** בדקתי שוב את
`bumpIvrAudioStatusOnUpload`/`bumpStatusOnUpload` (`ivr-audio.service.js:29-34`):
`if (idx < uploadIdx) return "הוקלט"; return currentStatus;` — לשורה
שכבר `"אושר"` (אינדקס 4), `4 < 2` הוא `false`, אז הסטטוס **נשאר** `"אושר"`,
לא יורד ל-`"הוקלט"`. כלומר התסמין המדויק חמור אפילו יותר ממה שתיארת:
לא "הסטטוס יורד וההקלטה יוצאת משימוש" — אלא **הסטטוס נשאר `"אושר"`
בעוד שהתוכן הפיזי מאחוריו כבר החליף** לקובץ חדש לא-נבדק, בלי שום חלון
בדיקה. ה-resolver (`row.status === APPROVED_STATUS`) ימשיך "לחשוב"
שהשורה תקינה ויגיש את הקובץ החדש-הלא-נבדק לשיחות מייד. התיקון שהצעתי
ב-9.11 (audioFile1/2 בלבד) לא פותר את זה — הוא רק דוחה את המחיקה
הפיזית, לא את הכניסה המיידית לשימוש. תיקון אמיתי צריך 3 מצבים בו-זמנית.

#### תשובה לנקודה 7 — כן, אפשרי עם 3 הסלוטים והסטטוס הקיימים, בלי סכמה חדשה

בתנאי שמגדירים מחדש את המשמעות של כל סלוט **רק לשורות `category="paymsg"`**
(אפס השפעה על 83+10 השורות הקיימות — הן ממשיכות בדיוק כמו היום, שני
הגיליונות הקיימים לא נוגעים בזה בכלל):

| סלוט | משמעות חדשה (paymsg בלבד) | מי קורא אותו |
|---|---|---|
| `audioFile1` | **פעיל בשיחות** | ה-resolver (`ivr-audio-resolver.service.js`) — **בלי שום שינוי קוד שם** |
| `audioFile2` | **גרסה קודמת** (רולבק צעד אחד) | רק ה-route החדש לשחזור |
| `audioFile3` | **ממתין לאישור** (הועלה, עבר בדיקת פורמט+המרה, טרם אושר) | רק אדמין (נגן/מחיקה — כפתורים גנריים קיימים) |
| `status` | ממשיך לתאר **רק** את `audioFile1` — משמעות זהה לחלוטין להיום | ה-resolver, בלי שינוי |

זה עובד כי ה-resolver כבר קורא **רק** `audioFile1`+`status` (אימתי
שוב, `ivr-audio-resolver.service.js:87-88`) — סלוטים 2/3 כבר לא נוגעים
בשום path של שיחה חיה, היום ובתמיד. **אין צורך בעמודה/טבלה חדשה.**

#### הזרימה המדויקת (עונה על נקודות 1-5)

**העלאה (חדש: לשורת paymsg, ה-route הקיים מחייב `slot=3` בלבד — דוחה
בקשת העלאה ל-slot 1/2 עבור `category="paymsg"` עם שגיאה ברורה, כדי
שאף קריאת API — לא רק ה-UI — לא תוכל לגעת ב-live בטעות):**
1. multer שומר את הקובץ; בדיקת פורמט + המרה אוטומטית (כמו סעיף 9.2,
   ללא שינוי בלוגיקה הטהורה) — **לתוך `audioFile3`**.
2. הצלחה: `UPDATE ... SET audioFile3=?, updatedAt=? WHERE audioId=?`
   — **`status` לא נכתב בכלל** (לא `bumpStatusOnUpload`, שום קריאה
   לפונקציה הזו בנתיב הזה). `audioFile1`/`status` הקיימים לא נוגעים בהם.
   → הגרסה הפעילה ממשיכה לשרת שיחות בלי שום הפרעה (נקודה 1).
3. כישלון (פורמט/המרה): כמו סעיף 9.2 — מנקים את מה שהועלה כרגע, לא
   נוגעים ב-`audioFile1/2/3` הקיימים בכלל, שגיאה בעברית (נקודה 3).

**אישור (נקודה 2) — לא כפתור חדש: משתמשים ב-route הקיים
`PUT /api/admin/ivr-audio/:id` (עדכון סטטוס), עם תוספת לוגיקה קטנה:**
כש-`body.status === "אושר"` **וגם** `existing.category === "paymsg"`
**וגם** `existing.audioFile3` לא ריק → זו לא "עדכון סטטוס רגיל", זו
**פעולת קידום אטומית**:
1. אימות: `audioFile3` קיים בפועל בדיסק ו-path-contained (לא רק
   שהעמודה לא ריקה).
2. `oldFile2 = existing.audioFile2` (גרסה-לפני-הקודמת) — נמחקת מהדיסק
   **אחרי** שה-UPDATE הבא מצליח (לא לפני — נקודה 5), ובכישלון מחיקה
   רק לוג, לא כישלון הפעולה.
3. `UPDATE ... SET audioFile2=audioFile1הישן, audioFile1=audioFile3הישן, audioFile3='', status='אושר', updatedAt=? WHERE audioId=?`
   — עדכון DB **אחד אטומי**. רק אחריו — השיחה הבאה מקבלת את הגרסה
   החדשה (נקודה 2, "משפיע מיד").
4. אם `audioFile3` ריק כשמאשרים (למשל אישור רגיל בלי גרסה ממתינה) —
   מתנהג בדיוק כמו היום (`updateIvrAudioRecording` רגיל, בלי סיבוב
   סלוטים) — **אפס שינוי התנהגות** לשורות הקיימות ולמקרה הרגיל.

**דחיית/מחיקת גרסה ממתינה:** ה-route הגנרי הקיים
`DELETE /api/admin/ivr-audio/:id/audio/3` — **בלי שום שינוי קוד** —
כבר עושה בדיוק את זה (מוחק קובץ + מנקה את העמודה).

**שחזור גרסה קודמת (נקודה 4) — כמו 9.11, עם האימותים המפורשים שביקשת:**
לפני ההחלפה: `audioFile2` לא ריק, `isPathContained` על שני השמות
(1 ו-2), `fs.existsSync` על שני הנתיבים בפועל. כל כשל → `400` עם הודעה
ברורה ("אין גרסה קודמת תקינה לשחזור"), **שום שינוי ב-DB**. הצלחה →
`UPDATE ... SET audioFile1=audioFile2הישן, audioFile2=audioFile1הישן ...`
(swap אטומי אחד) — לא נוגעים בדיסק בכלל (שני הקבצים כבר שם).

#### הגנת concurrency (נקודה 6)

`Map` פשוט בזיכרון התהליך, `audioId -> true` בזמן שפעולת
upload/approve/restore **על אותו audioId** רצה. בקשה שנייה לאותו
audioId בזמן שהראשונה עדיין רצה → `409` מיידי ("פעולה אחרת כבר
מתבצעת על הקלטה זו, נסה שוב בעוד רגע"), לא נכנסת ללוגיקה בכלל. מספיק
כי כל שלוש הפעולות סינכרוניות מקצה לקצה (`execFileSync`/`fs.*Sync`,
בלי `await` באמצע) — התהליך היחיד של Node לא "מתחלף" באמצע ריצה של
אחת מהן ממילא (הבדיקה הזו רק מונעת תור-בקשות מצטבר, לא race אמיתי בתוך
תהליך אחד). לא בודק/פותר race בין כמה תהליכי pm2 — הקוד הקיים כולו
(כל `db.js`) כבר מניח תהליך יחיד (`fork` mode, לא `cluster`), אין בזה
סיכון חדש שהתוכנית הזו מוסיפה.

#### רשימת שינויים מינימלית (מעודכנת, מחליפה את 9.3+9.11)

- **`server.js`** — ה-upload route: ענף חדש ל-`category==="paymsg"`
  (כופה slot=3, לא קורא ל-`bumpStatusOnUpload`). ה-status-PUT route:
  ענף חדש לזיהוי "אישור עם גרסה ממתינה" (סיבוב סלוטים אטומי). route
  חדש אחד: `restore-previous`. Map בזיכרון ל-concurrency guard.
- **ללא שינוי בכלל:** `db.js` (אין עמודה/טבלה חדשה, `setIvrAudioRecordingFileSlot`
  משמש כמו שהוא — רק בוחרים איזה `status` מעבירים לו), ה-resolver, ה-DELETE
  route הקיים, שני הגיליונות/83+10 השורות הקיימות (ההתנהגות שלהן
  משתנה **אפס** — הענפים החדשים מותנים תמיד ב-`category==="paymsg"`).
